Writing this test surfaced a **pre-existing, previously-undetected break in the e2e Jest config**, not something this task introduced: `packages/shared`'s `package.json` has `"type": "module"`, so once `CoinsModule`'s DTOs started importing `@coin-collector/shared` (back in 4.1/4.2), every e2e spec that boots `AppModule` — including `auth.e2e-spec.ts` and `user-sets.e2e-spec.ts`, both previously marked done — silently started failing at the Jest level (`SyntaxError: Unexpected token 'export'`). It went unnoticed because 4.2–4.4's own verification was done via live `curl` against the dev server, never `pnpm --filter api test:e2e`. Root cause: TypeScript's `NodeNext` module setting (from the shared `tsconfig.base.json`) emits real ESM for any file under a `"type": "module"` package regardless of what the *consumer* is — that's correct for production (Node 22's `require(esm)` support is why the live API never had a problem) but Jest's CommonJS-only runtime can't execute that output at all, and no amount of `ts-jest` config overrides (`isolatedModules`, inline `tsconfig: { module: "CommonJS" }`) actually changes the emitted syntax, because `ts-jest` resolves the module kind from the discovered project config keyed off the file's own package, not the override. Fixed in `apps/api/test/jest-e2e.json` with a dedicated transform entry routing anything under `packages/shared/src/*.ts` through a small custom transformer (`apps/api/test/support/shared-esm-transformer.js`, a direct `ts.transpileModule` call forcing `module: CommonJS`) instead of the default `ts-jest` entry, plus `moduleNameMapper` entries pointing `@coin-collector/shared` and its two internal relative imports (`./enums.js`, `./contracts.js`) at the package's TS sources rather than its built ESM `dist` — test-only, the real build/deploy path is untouched. ## What this project is (v2)

Coin Collector Companion — a personal coin-catalog and collection tracker. Core loop (PRD v2): browse an open coin catalog (anonymously or logged in), build a set — by filtering the catalog, by manual pick, or by cloning a canonical set or another user's public set — mark coins owned/not-owned (a single ownership record per `(user, coin)`, global across every set that coin appears in, not per-set), and see a gap list + completion % for that set. The **gap view** remains the core differentiator; everything else (catalog browsing, set cloning, sharing) exists to feed coins into it. Read [docs/prd_v2.md](docs/prd_v2.md) (supersedes `docs/prd.md`) before making scope decisions — it resolves several things explicitly and explains *why*, not just *what*.

## Architecture (as planned, v2)

Frontend (Next.js, App Router) and backend (NestJS) run as **separate services at runtime** — not a Next.js app using API routes as the backend — this is deliberate, to force a real API boundary (DTOs, validation, guards) for portfolio purposes. Do not collapse them into one running service.

- **Repo layout:** single monorepo (pnpm workspaces: `apps/web`, `apps/api`, `packages/shared` for DTO/type sharing), plus `scripts/import-catalog/` (offline catalog import, not part of the API's request path) and `seed/templates/` (versioned JSON canonical-set definitions) — see [docs/system-design_v2.md](docs/system-design_v2.md) §2.1.
- **DB:** PostgreSQL via Prisma (decided — do not reintroduce a TypeORM debate).
- **Auth:** hand-rolled email/password + JWT (bcrypt + Passport JWT strategy) — the same module built under v1, restored from git history rather than rebuilt from scratch (see the 2026-07-19 auth-restore changelog entry below). Neon Auth (Stack Auth under the hood, JWKS/EdDSA verification) was the original v2 plan, but its Beta status meant the originally-planned time-boxed spike ([docs/build-roadmap.md](docs/build-roadmap.md) Week 1) was skipped outright in favor of this proven, self-contained mechanism — see [docs/system-design_v2.md](docs/system-design_v2.md) §4.2/§6 for the full trade-off. This means the v2 data model has its own `users` table again (§4.1), not one owned externally by Neon Auth.
- **Catalog images:** hotlinked from Wikimedia Commons at import time (`coins.image_url` stores Commons' own URL) — no re-hosting, no object storage for catalog images. Distinct from **user-uploaded photos of owned coins**, which stay out of scope until a later phase (Cloudflare R2, presigned uploads) — don't build upload infrastructure for either before it's actually scheduled.
- **Hosting (when deploying):** Vercel (frontend), Render (backend), Neon (DB). Free tier throughout; Render cold-starts after 15 min idle, Neon autosuspends similarly — both accepted, documented tradeoffs, not bugs to fix.

## Data model (v2 — see [docs/system-design_v2.md](docs/system-design_v2.md) §4.1 for full detail)

This **replaces** the v1 model (`User`/`CollectionSet`/`SetSlot`/`UserSet`/`CoinItem`) described in the historical Project status log above; as of this writing that migration hasn't landed in code yet.

- `coins` — the catalog itself: one row per distinct `(country, denomination, year, mint_mark, variety)`, populated by an offline import script from open data (Wikipedia/US Mint/Wikimedia Commons), never fetched live. `mint_mark`/`variety` are `NOT NULL` with `''` as the default (not nullable) — Postgres NULL-distinctness would otherwise defeat the composite uniqueness constraint, the same class of bug v1's `SetSlot` seeding had to work around in script code; v2 closes it with a non-null default instead.
- `canonical_sets` / `canonical_set_coins` — admin-curated sets (e.g. US Presidents), loaded from versioned JSON templates in `seed/templates/` — same versioned-template philosophy as v1's set-slot seeding, just against the new schema.
- `user_sets` / `user_set_coins` — a user's own sets: built from scratch, filtered from the catalog, or cloned from a canonical set or another user's public set (`cloned_from_canonical_id`/`cloned_from_user_set_id`, both nullable and `onDelete: SetNull` — a clone survives its source being deleted). **Every `user_sets` row is public and cloneable by default in v1** — no per-set privacy control yet (post-MVP: an `is_public` flag).
- `ownership` — existence-based, not a boolean column: a `(user_id, coin_id)` row means owned, `owned_at` is metadata on it. Owned is global to the user across every set the coin appears in — same "global ownership, not per-set" principle v1's `CoinItem` already had, just modeled as its own table now instead of a FK-bearing row per slot. No relationship to `user_sets`/`user_set_coins` at all (joined only incidentally via `coin_id`) — deleting a set must never cascade into `ownership`.

Mint-mark convention carries over conceptually from v1 but the representation changed: **no mint mark is `''` in `coins`** (not `null` — that was the v1/`SetSlot` convention, this is a different table with a different NULL-handling story), sanitized to that literal at import time regardless of what placeholder the source used (`null`, `"None"`, `"N/A"`, whitespace, etc.).

Gap computation: for a given `user_set`, gap = the set's coins with no matching `ownership` row for that user — a left join / anti-join, not a stored flag. Keep this as a simple query, same as v1 — it's the core product, don't over-engineer it.

## Scope discipline (v2)

[docs/prd_v2.md](docs/prd_v2.md) and [docs/system-design_v2.md](docs/system-design_v2.md) are organized around Requirements, not the v1 Phase 1–4 structure — check both before adding a feature:

- **No theme/subject tags** — cut from v1 entirely, not deferred. No `tags`/`coin_tags` tables. A themed set ("coins depicting animals") is still buildable via manual pick or cloning another user's public set.
- **No `is_public` flag / per-set privacy** — every `user_sets` row is public and cloneable in v1; this is post-MVP.
- **No mid-set drag-and-drop reordering** — `PATCH /sets/:id/coins` appends at `max(position) + 1` only; no re-indexing/fractional-position scheme in v1.
- **No Numista dependency** — open data only (Wikipedia/US Mint/Wikimedia Commons) for v1's catalog. Numista is optional, permission-gated future enrichment, never a v1 blocker.
- **No image re-hosting** — hotlink Commons URLs directly; don't build a download/object-storage pipeline for catalog images before it's an actual problem (broken/stale links in practice).
- **Canonical sets are required, not eligible for descoping** — the Success Metrics need a canonical-set-derived set to complete the core loop end-to-end, same as a filter-built set. If catalog work runs long, cut the seed-authoring *tooling* first (fall back to one hardcoded canonical set), never canonical sets themselves.
- The core-loop acceptance test is the scope firewall (PRD v2 Success Metrics): create/edit a set, add coins (filter-built and canonical-clone paths both), mark owned/not-owned, view gaps and completion % — end to end, without bugs. Anything not needed for that path waits.
