## Day 1 — Sets module: create, rename, delete, list

- [x] 1.1 `SetsModule`/`SetsController`/`SetsService` (`apps/api/src/sets`): `GET /sets` (current user's own sets, required auth, unpaginated per SD §3 — genuinely per-user and bounded). `PATCH /sets/:id` (rename/metadata, owner-only — 404 unknown, 403 foreign). `DELETE /sets/:id` (owner-only; cascades to `user_set_coins` only via the schema's `onDelete: Cascade` on `userSetId` — must **never** touch `ownership`, SD §4.1).
- [x] 1.2 `POST /sets`: three shapes resolving to one handler — blank (`{ name }`), `{ cloneFrom: { type: 'canonical', id } }`, `{ cloneFrom: { type: 'user', id } }` (SD §3, §4.6). Cloning copies the source's ordered coin list verbatim into new `user_set_coins` rows (positions carried over, not renumbered) and sets `clonedFromCanonicalId`/`clonedFromUserSetId` accordingly — exactly one of the two, never both, enforced by the handler (not a DB constraint, per SD §4.1's note that this is a low-stakes invariant the API already guarantees by construction). Clone-from-canonical works against any `canonical_sets` row (all public); clone-from-user works against **any** `user_sets` row, not just the caller's own, since every set is public in v1 (SD §4.6) — there is no ownership or visibility check on the *source* of a clone, only on mutating the *result*.
- [x] 1.3 `ParseUUIDPipe` on every `:id` param in this controller from the start, not added later — SD §3's route-ordering gotcha note explains why this matters once `canonical`/`public` literal routes land beside `:id` on Day 2 (a stray `"canonical"` or `"public"` falling through to `:id` should 400 loud, not 404 quiet).
- [x] 1.4 Manual pass: create a blank set, rename it, delete it (confirm `user_set_coins` cascades — table's empty anyway at this point, real content isn't added until Day 2's `PATCH /sets/:id/coins`); clone a throwaway canonical-set row (seed one manually via Prisma Studio/`psql` if Day 3's real seed script isn't built yet) and confirm the clone's coins/positions match the source and `clonedFromCanonicalId` is set; a second user attempting `PATCH`/`DELETE` on the first user's set gets 403; unauthenticated `GET /sets` gets 401.

## Day 2 — Coin membership, canonical-set reads, public-set reads

- [x] 2.1 `PATCH /sets/:id/coins` (owner-only): add/remove coin IDs. Adds append at `max(position) + 1` within that set (`1` if empty) — no mid-set insertion, no reordering (SD §4.1, explicit MVP scope cut). Written as skip-on-conflict against `@@unique([userSetId, coinId])` (Prisma `createMany({ skipDuplicates: true })` or equivalent) so a double-click, a retried request, or an overlapping filter re-add is a no-op, not a duplicate row or a 500 — this is the constraint that keeps `GET /sets/:id/gaps`' completion % honest (SD §4.1). Removes are a plain delete of the join row(s); removing a coin never touches `ownership`. Deviation: the `add` array is also de-duped in the service (`[...new Set(dto.add)]`) before the `createMany` call, on top of `skipDuplicates` — belt-and-suspenders against a same-array duplicate rather than relying solely on Postgres's `ON CONFLICT DO NOTHING` ordering; response body is the set's current `UserSetCoin` rows (flat join-row shape), since SD §3 didn't pin down an exact response shape for this endpoint.
- [x] 2.2 `GET /sets/canonical` (list, no auth) and `GET /sets/canonical/:id` (detail + ordered coin list, no auth).
- [x] 2.3 `GET /sets/public` (list every `user_sets` row across all users — all of them, since none are private yet — paginated `page`/`limit`, reusing the same pagination DTO/shape as `GET /catalog`, SD §3/§4.6) and `GET /sets/public/:id` (detail + coin list, no auth, no owner display name or other user-identifying detail — SD §4.6 is explicit that public profiles are out of scope).
- [x] 2.4 **Register the literal-segment routes before `:id`.** `GET /sets/canonical` and `GET /sets/public` must be declared above `GET /sets/:id/...` in `SetsController` — NestJS matches in declaration order, and getting this backwards silently captures `"canonical"`/`"public"` as `:id = "canonical"` instead of hitting the literal route (SD §3). Verify by hitting both literal routes and a `:id` route in the same manual pass, not by code inspection alone.
- [x] 2.5 Manual pass: `PATCH /sets/:id/coins` adding the same coin ID twice in one call, and again via two separate repeated calls, then confirm the set's coin count reflects one slot, not two or three (SD §4.1's dedup-corrupts-completion-% concern, verified directly rather than assumed from the code). `GET /sets/canonical`/`GET /sets/public` both return correctly with no `Authorization` header; `GET /sets/canonical` isn't shadowed by a `:id` handler (2.4). Also verified live: add→remove→re-add produces no stale position collision; all four new read endpoints 200 with no `Authorization` header; `GET /sets/canonical/:id` 404s on an unknown id; `GET /sets/public` paginates 3 throwaway rows across two pages (limit 2) with no overlap/gap; `PATCH /sets/:id/coins` 401s unauthenticated and 403s a non-owner. Throwaway users/sets created for this pass were deleted afterward (DB row counts confirmed back to baseline).

## Day 4 — Collection/ownership + gap view

- [x] 4.1 `CollectionModule`: `PATCH /collection/:coinId` with an explicit `{ owned: boolean }` body (required auth) — idempotent, not a toggle (SD §2.3, §4.1). `true` upserts the `(userId, coinId)` row into `ownership`; `false` deletes it if present. Both directions are no-ops on an already-matching state (marking an already-owned coin owned again, or unmarking an already-absent one) — verified at the unit level (mocked Prisma) that `owned: true` calls `upsert({ update: {} })`, which writes nothing on conflict, and `owned: false` calls `deleteMany` (never throws on zero matches). Deviations: (1) `update: {}` means `ownedAt` is preserved, not reset, on a repeated `owned: true` — chosen because `ownedAt` is documented (SD §4.1) as "when it was marked owned," not "when it was last confirmed owned"; (2) an unknown `coinId` is mapped from Prisma's `P2003` foreign-key-violation error to a `404 Not Found`, mirroring the existing `P2002`→409 pattern in `AuthService.register`, rather than surfacing a raw 500 — SD §3 doesn't pin either decision down, so both are recorded here rather than left implicit.
- [x] 4.2 `GET /collection` (required auth): the caller's full ownership list, filterable by country/year (PRD requirement 5/6). Filters apply through the `coin` relation (`Ownership` itself has no `country`/`year` columns) via `FindCollectionQueryDto`, mirroring `FindCatalogQueryDto`'s conventions minus pagination.
- [x] 4.3 `GET /sets/:id/gaps` (required auth, **not** owner-restricted — SD §4.6): the requested set's coin list left-joined against `ownership WHERE userId = <caller>`, with a computed `ownedCount`/`totalCount` and completion %. Works identically for the caller's own sets and any public set belonging to someone else — the response reflects the caller's own ownership regardless of who owns the set (this is what lets a user preview their match against someone else's set before cloning it, SD §4.6). One query (`userSet.findUnique` with a nested `coin.ownerships` include filtered to the caller), no per-slot loop (mirrors the v1 gap-view precedent already noted in `CLAUDE.md`); `completionPercent` guards the zero-coin case explicitly rather than dividing by zero.
- [ ] 4.4 Manual pass — set-deletion/ownership isolation: mark a coin owned within a set, `DELETE` that set, then confirm via `GET /collection` that the coin still shows owned (SD §4.1's explicit concern: `DELETE /sets/:id` must structurally be unable to cascade into `ownership`, since there's no FK path there at all — confirm the behavior, don't just trust the schema). **Pending manual pass** — requires the live dev DB; not run by the automated pipeline that built 4.1–4.3 (mocked-Prisma unit tests cover the underlying logic but cannot observe real Postgres cascade behavior).
- [ ] 4.5 Manual pass — clone-lineage isolation: user A creates and populates a public set, user B clones it (`POST /sets` with `cloneFrom: { type: 'user', id }`), then user A's original set is deleted. Confirm user B's clone still exists with its coins intact and `clonedFromUserSetId` reset to `null` (not cascaded away) — this is the `onDelete: SetNull` behavior from Day 2's schema, confirmed live rather than assumed from the migration. **Pending manual pass** — same reason as 4.4; also covers the idempotency/filter/gap-view-on-non-owned-set/401 checks called out alongside it in this run's task description.
