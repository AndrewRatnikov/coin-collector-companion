No test framework is configured in `apps/web` (noted already in `CLAUDE.md`'s v1 log) — every manual pass this week is a real browser/Playwright check against running dev servers, same as Weeks 1–2's "Manual pass" items, not an automated suite.

## Day 4 — Set editor + gap view + collection page

## Explicitly NOT this week (resist)

- Any backend/API change — Weeks 1–2 already closed the entire API surface; if a UI need surfaces a real API gap, that's a scope bug in Week 1/2's plan to flag, not something to quietly patch in from the frontend this week.
- `is_public` flag / per-set privacy UI — post-MVP (PRD Requirement 18, SD §4.6); every set stays public in the UI too, matching the backend.
- Mid-set drag-and-drop reordering in the editor — `PATCH /sets/:id/coins` stays append-only server-side (SD §4.1); the editor UI shouldn't imply reordering is possible.
- Theme/subject tag filtering/browsing — cut from v1 entirely, not deferred (PRD Requirement 15); the catalog filter form is exactly the SD §3.1 fields, nothing more.
- Public user profiles / any per-user identifying detail beyond the opaque set-owner relationship already exposed by the API — explicitly out of scope (PRD Non-Goals, SD §4.6).
- Numista integration, CSV import, analytics/value-tracking — all post-MVP per the PRD's "v2 / Post-MVP Direction" section; nothing this week should assume any of that data exists.
- Deploy verification against Vercel/Render prod — Buffer week (`build-roadmap.md`), not this week; Week 3 is local-dev-only.
